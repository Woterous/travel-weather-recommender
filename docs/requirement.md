# 项目名称

基于天气数据的旅游出行推荐与可视化系统

## 文档目的

本文件用于冻结项目 V1 的需求边界、技术基线和验收口径，避免后续实现阶段出现范围膨胀、技术漂移和功能失控。

本文档回答 4 个问题：

1. 项目要做什么
2. 项目明确不做什么
3. 项目采用什么技术和数据策略
4. 做到什么程度才算达到课程设计可验收状态

---

## 1. 项目定位

本项目是一个面向课程设计演示的“旅游天气推荐系统”，核心价值是：

- 采集多个旅游城市的天气数据
- 对数据进行清洗和标准化
- 使用规则型评分算法给出旅游适宜度推荐
- 通过网页可视化展示排行榜、城市详情、城市对比和历史分析

本项目不是综合旅游平台，不是通用天气 App，也不是智能问答系统。

---

## 2. 项目成功标准

满足以下条件时，视为项目达到 V1 成功标准：

- 可以完整演示“数据获取 -> 数据清洗 -> 评分计算 -> 推荐展示”的闭环
- 支持 10 个固定城市
- 推荐结果会随着用户偏好变化而变化
- 首页直接展示核心推荐结果和主要功能入口
- 前端能够完成排行榜、详情、对比、历史分析展示
- 系统逻辑可以在 5 分钟内向老师讲清楚
- 项目本地可运行，不依赖复杂部署环境

---

## 3. 已冻结实施基线

### 3.1 技术栈决策

首版固定采用以下技术方案：

- 后端：Python + Flask
- 数据处理：pandas
- 抓取与解析：requests + BeautifulSoup4
- 前端：HTML + CSS + JavaScript + ECharts
- 主存储：SQLite
- 原始抓取备份：JSON 或 CSV

首版明确不引入：

- Spark
- Hive
- Hadoop
- Vue
- React
- MySQL
- Redis
- 容器编排和复杂分布式部署

### 3.2 数据获取决策

- 数据获取以公开网页抓取为主
- 公开 API 仅作为补充来源
- 样例数据仅用于开发调试，不作为默认验收数据
- 所有正式展示结果应优先来自真实抓取和清洗后的数据
- 抓取链路与页面展示链路必须分离
- 页面访问默认只读取本地 SQLite，不以“打开页面”触发实时抓取
- 未来天气通过手动刷新或定时刷新更新
- 历史月度统计数据允许长期存储在本地
- 页面需要展示最近一次成功更新时间

### 3.3 城市范围决策

首版固定城市为以下 10 个：

- 北京
- 上海
- 杭州
- 南京
- 成都
- 重庆
- 厦门
- 青岛
- 西安
- 三亚

首版不做任意城市自由扩展。

### 3.4 历史数据决策

- 首版历史数据以月度统计为主
- 不强制实现全量日级明细
- 历史分析围绕月度统计指标展开

### 3.5 AQI 决策

- AQI 是增强项，不是首版硬依赖
- 如果数据源稳定，可启用 AQI 维度
- 如果数据源不稳定，系统必须进入“无 AQI 降级模式”
- 无 AQI 模式下，不允许展示 AQI 分项，也不允许保留 AQI 权重

### 3.6 首页决策

- 首页与排行榜页合并
- 首页直接展示核心推荐结果、排行榜和主要入口
- 不单独开发一个纯介绍型首页

## 4. 范围边界

### 4.1 V1 必做范围

- 固定城市列表的未来天气采集
- 固定城市列表的历史月度统计数据整理
- 原始数据保存
- 清洗后标准数据生成
- 规则型旅游适宜度评分
- 偏好驱动的权重或规则调整
- 首页兼排行榜页
- 单城市详情页
- 双城市对比页
- 至少 2 个历史分析指标
- 基于 Flask 的网页演示

### 4.2 V1 明确不做

- 用户注册、登录、权限管理
- 酒店预订、机票预订、门票购买
- 地图导航、路径规划、实时定位
- 评论、点赞、收藏、社交分享
- 聊天机器人、大模型问答
- 深度学习模型训练
- 实时流式计算
- 分布式集群部署
- 移动端 App
- 任意城市自由搜索并实时扩展数据

### 4.3 容易失控的边界说明

- “城市范围”不是全国任意城市，而是固定 10 城市
- “城市对比”首版只支持 2 个城市，不做多城市自由扩展
- “天气预测”首版只覆盖数据源可稳定提供的近未来日期，默认不超过 7 天
- “历史分析”首版只做 2 到 3 个可解释指标，不做复杂预测模型
- “推荐理由”必须来自规则和分项得分，不允许生成无法追溯依据的文案

---

## 5. 功能需求

### 5.1 数据采集模块

必须支持：

- 对固定 10 城市进行批量采集
- 获取未来天气数据
- 获取历史月度统计数据或可支撑月度统计的基础数据
- 将原始数据落盘
- 将抓取逻辑与解析逻辑分开
- 将清洗后的展示数据写入本地 SQLite

V1 的硬限制：

- 城市列表必须配置化，不能硬编码在多个文件中
- 未来天气默认只做近 7 天内可稳定抓取的数据
- 历史数据首版只要求支撑月度统计，不要求完整日级明细
- 数据源以公开网页为主，公开 API 只作补充
- 页面请求不直接调用抓取逻辑
- 历史数据默认从本地持久化结果读取
- 未来天气默认优先读取本地缓存结果，按刷新策略更新

### 5.1.1 刷新与缓存策略

首版必须采用以下展示策略：

- 页面打开时默认读取本地 SQLite
- 不要求每次访问页面时实时抓取
- 未来天气支持“手动刷新”或“定时刷新”两种方式，至少实现其中一种
- 历史月度统计数据默认本地长期保存，不重复高频抓取
- 当抓取失败时，页面应继续展示最近一次成功抓取的数据，并显示更新时间和状态说明

### 5.2 数据清洗模块

必须支持：

- 去重
- 缺失值处理
- 温度字段数值化
- 日期字段统一格式
- 天气描述标准化
- 风力字段标准化
- 历史与未来数据字段映射到统一结构

输出结果必须可直接供评分和前端展示使用。

### 5.3 推荐评分模块

必须支持：

- 单城市单日期总分计算
- 分项得分计算
- 权重展示
- 推荐理由生成

评分方式限定为规则型，不做机器学习训练。

### 5.4 偏好设置模块

至少支持以下 4 类偏好：

- 降雨敏感度
- 温度偏好
- 风力敏感度
- 旅游风格

空气质量敏感度为可选增强项，仅在 AQI 可稳定获取时启用。

偏好必须真实影响结果，至少满足以下要求：

- 修改偏好后，总分会变化
- 至少 2 个分项权重或评分规则发生变化
- 前端可展示“当前偏好如何影响评分”

### 5.5 首页兼排行榜模块

必须支持：

- 指定日期的城市推荐排行榜
- 按总分排序
- 展示总分、分项摘要和推荐理由
- 偏好变化后刷新结果
- 展示主要功能入口，包括详情、对比、历史分析和偏好设置
- 展示当前数据最近更新时间
- 展示当前数据是否来自本地缓存或最近刷新结果

明确限制：

- 排行榜只在固定 10 城市范围内比较
- 不做“全国城市实时排行榜”

### 5.6 城市详情模块

必须展示：

- 基础天气信息
- 总分和分项得分
- 推荐理由
- 至少 1 个历史分析结果
- 至少 1 个图表

### 5.7 城市对比模块

V1 只要求支持 2 个城市对比，必须包含：

- 总分对比
- 温度对比
- 降雨对比
- 风力对比
- 至少 1 个历史指标对比

### 5.8 历史分析模块

至少实现以下 2 项：

- 历史平均适宜度
- 历史雨天频率
- 历史舒适温度天数占比
- 历史天气稳定性

要求：

- 必须有真实计算逻辑
- 不允许只写静态描述
- 展示维度以城市和月份为主

### 5.9 前端展示模块

必须具备以下页面或等价页面：

- 首页兼排行榜页
- 偏好设置页
- 城市详情页
- 城市对比页
- 历史分析页

必须具备以下交互：

- 日期选择
- 城市选择
- 偏好调整
- 查看详情
- 查看对比

## 6. 数据标准

### 6.1 清洗后标准字段

以下字段为必备字段：

| 字段名 | 含义 |
| --- | --- |
| city | 城市名 |
| date | 目标日期 |
| max_temp | 最高温 |
| min_temp | 最低温 |
| avg_temp | 平均温 |
| weather_type | 标准化天气类型 |
| wind_level | 标准化风力等级 |
| rain_flag | 是否降雨 |
| source_type | 数据类型，取值如 forecast 或 history_monthly |
| source_name | 来源名称 |
| crawl_time | 抓取时间 |

以下字段为可选字段：

| 字段名 | 含义 |
| --- | --- |
| aqi | 空气质量指数 |
| rain_text | 原始降雨描述 |
| score_total | 总分 |
| score_detail | 分项得分结构 |

### 6.2 字段原则

- 字段命名统一使用英文
- 温度统一为数值型，单位固定为摄氏度
- 日期统一为 `YYYY-MM-DD`
- `weather_type` 必须做归一化映射，如晴、多云、阴、雨、雪、雾霾等
- `rain_flag` 必须有明确生成规则，不能凭主观填写

## 7. 评分与偏好规则

### 7.1 必备评分维度

V1 必做 5 个维度：

- 温度舒适度
- 降雨风险
- 风力舒适度
- 天气类型
- 历史稳定性

AQI 作为可选第 6 维，仅在数据稳定时启用。

### 7.2 权重规则

无 AQI 时，默认权重建议为：

- 温度舒适度：0.30
- 降雨风险：0.25
- 风力舒适度：0.10
- 天气类型：0.10
- 历史稳定性：0.25

有 AQI 时，默认权重建议为：

- 温度舒适度：0.25
- 降雨风险：0.25
- 风力舒适度：0.10
- 天气类型：0.10
- 历史稳定性：0.20
- AQI：0.10

### 7.3 偏好调整约束

- 偏好调整后，权重总和必须仍为 1.00
- 单一维度权重不得被调整到 0.40 以上
- 单一维度权重不得低于 0.05
- 缺失 AQI 数据时，AQI 权重必须移除并重新归一化其余维度

### 7.4 推荐理由约束

推荐理由必须可追溯到以下内容：

- 最高得分项
- 最低得分项
- 至少 1 个用户偏好影响
- 至少 1 个天气或历史事实

不允许使用“综合来看非常适合”这类无证据文本作为唯一理由。

## 8. 非功能需求

- 本地一台普通电脑可运行
- 首次演示不依赖实时外网抓取也可以完成，但展示数据必须来自此前真实抓取结果
- 页面首屏展示不应被实时抓取阻塞
- 数据缺失时页面不崩溃，能显示降级提示
- 采集失败时有失败信息，不允许静默失败
- 页面主要面向课堂演示，优先保证信息清楚而不是炫技
- 代码结构应支持老师查看模块划分

- `data/raw/debug_samples/` 只用于开发调试，不作为默认验收数据
- `scripts/` 用于手动执行抓取和生成处理数据

## 10. 风险与降级策略

### 10.1 数据源风险

如果站点结构变化、访问失败或字段缺失，允许采用以下降级方案：

- 更换同类公开网页来源
- 使用公开 API 补充缺失字段
- 将 AQI 模块整体降级为“暂不支持”
- 将历史分析严格收口到月度统计
- 保留并继续展示最近一次成功抓取的本地数据

不允许采用以下错误降级方式：

- 前端直接硬编码假排行榜
- 伪造历史统计结果
- 将抓不到的数据默认为“晴天”或“无雨”
- 用调试样例数据冒充正式验收数据

### 10.2 范围风险

以下情况一律视为范围失控：

- 临时加入登录、地图、预订等无关模块
- 临时扩展成任意城市搜索系统
- 临时换成前后端分离大框架
- 在没有需求更新的情况下引入大数据平台或模型训练

---

## 11. 验收口径

项目通过验收，至少需要满足：

- 数据链路真实存在，不是静态假页面
- 推荐结果由真实算法计算得出
- 偏好会改变结果
- 历史分析有真实计算，且以月度统计为主
- 页面可展示首页兼排行榜、详情、对比、历史分析
- 文档能说明数据来源、字段、算法和降级策略
- 工程结构符合本文件定义的模块边界
