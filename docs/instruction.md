# 项目开发执行说明

基于天气数据的旅游出行推荐与可视化系统

## 1. 文档用途

本文件不是需求说明，而是后续开发时必须遵守的执行规范。

目标只有一个：

- 让开发过程始终围绕课程设计的最小可交付版本推进
- 防止 AI 或开发者擅自扩范围、换技术、堆功能

---

## 2. 执行优先级

后续开发时，优先级按以下顺序执行：

1. 用户明确确认的最新决定
2. `docs/requirement.md`
3. `docs/checklist.md`
4. 本文件

如果这 3 份文档互相冲突，以 `requirement.md` 为准，并先修正文档，再继续开发。

---

## 3. 已冻结开发基线

后续开发不得再对以下基线做默认改动：

- 技术栈固定为 Python + Flask + pandas
- 主存储固定为 SQLite，原始抓取可保留 JSON 或 CSV 备份
- 不引入 Spark、Hive、Hadoop
- 数据获取以公开网页抓取为主，公开 API 仅作补充
- 样例数据仅用于开发调试
- 城市固定为北京、上海、杭州、南京、成都、重庆、厦门、青岛、西安、三亚
- 历史数据首版以月度统计为主
- AQI 为增强项，允许无 AQI 降级模式
- 首页与排行榜页合并

如果后续需要改动，先改 `requirement.md`，再改代码。

---

## 4. 项目总原则

### 4.1 优先级原则

后续所有实现都必须服从以下优先级：

1. 可完成
2. 可解释
3. 可演示
4. 可维护
5. 再考虑美观和扩展

### 4.2 默认实现原则

- 优先用最小可运行方案
- 优先用规则型、可解释方案
- 优先用本地部署方案
- 优先减少依赖数量
- 优先做出完整闭环，而不是局部做得很花

---

## 5. 明确禁止事项

未经用户明确同意，不允许新增以下内容：

- 登录注册
- 用户权限
- 酒店、机票、门票等预订功能
- 地图、导航、路线规划
- 评论、收藏、社区
- AI 聊天或大模型问答
- 深度学习推荐模型
- 实时消息推送
- App、小程序、前后端多端同步
- 任意城市实时搜索扩展

未经用户明确同意，不允许擅自改为以下技术方案：

- Vue
- React
- 前后端完全分离架构
- MySQL
- Redis
- Docker 复杂部署
- Spark / Hadoop / Hive

如果确实需要改动，先改需求文档，再动代码。

---

## 6. 数据源执行规则

### 6.1 合法性与稳定性

- 只使用公开可访问的数据
- 不绕过验证码、登录、付费墙
- 不使用明显违反站点规则的抓取方式
- 不依赖高频抓取

### 6.2 开发策略

- 先做一个稳定网页来源跑通
- 再考虑是否增加公开 API 作为补充
- 未来天气、历史月度统计、AQI 可以来自不同来源
- 每个来源都必须能说明字段来源和抓取时间
- 样例数据只允许放在调试目录，不进入正式展示主链路
- 页面展示默认从本地 SQLite 读取，不直接读抓取页面
- 抓取完成后必须先落盘、清洗，再进入页面展示链路

### 6.3 降级规则

如果数据源不稳定，按以下顺序降级：

1. 更换同类公开网页来源
2. 用公开 API 补充个别字段
3. 降低字段要求，例如暂时取消 AQI

不允许：

- 为了演示方便，前端直接硬编码假排行榜
- 伪造历史统计结果
- 将抓不到的数据默认为“晴天”或“无雨”
- 用调试样例数据代替正式演示数据且不加说明
- 把“访问页面”直接实现成一次实时抓取

### 6.4 刷新与缓存规则

- 页面访问默认只读本地 SQLite
- 历史月度统计数据默认长期保存在本地
- 未来天气数据通过手动刷新或定时任务更新，不随每次页面访问更新
- 抓取失败时保留上一次成功数据，不清空现有展示结果
- 页面必须显示最近一次成功抓取时间
- 如果当前展示的是缓存数据，页面应有明确说明

---

## 7. 功能实现边界

### 7.1 城市范围

- 首版只支持固定 10 城市
- 不做任意城市即时扩展

### 7.2 时间范围

- 未来天气默认只做近 7 天
- 历史分析只做课程设计够用的 2 到 3 个指标
- 历史维度以月度统计为主
- 不做长期趋势预测

### 7.3 对比范围

- 首版只支持 2 个城市对比
- 不做多城市复杂联动分析

### 7.4 推荐算法范围

- 只做规则型算法
- 分数必须可解释
- 推荐理由必须能映射到分项得分
- 偏好必须真实影响结果

---

## 8. 项目结构执行规则

后续开发必须遵守以下目录边界：

- `config/` 只放配置，不放业务计算
- `crawler/` 只放抓取与解析，不放页面逻辑
- `service/` 只放清洗、评分、排行、对比、历史分析逻辑
- `web/` 只放 Flask 路由、模板和静态资源
- `data/raw/debug_samples/` 只放调试样例数据
- `scripts/` 只放手动执行入口

开发时必须遵守：

- 不要把爬虫、清洗、评分和前端逻辑写在一个文件中
- 城市列表、偏好默认值、数据源地址必须配置化
- 页面模板命名与页面职责保持一致
- 路由层只负责组织输入输出，不承担核心评分计算
- 路由层不直接触发实时抓取作为默认展示流程

---

## 9. 推荐开发顺序

后续开发请严格按以下顺序推进：

### 阶段 0：边界冻结

- 固定城市名单
- 固定数据源方案
- 固定历史月度统计口径
- 固定 AQI 降级策略
- 固定项目目录结构

### 阶段 1：数据链路跑通

- 获取未来天气原始数据
- 获取历史月度统计原始数据
- 原始数据落盘
- 输出清洗后的标准数据
- 将展示用结果写入 SQLite

阶段 1 未完成前，不进入复杂页面开发。

### 阶段 2：算法跑通

- 实现基础评分
- 实现偏好调整
- 输出总分、分项得分、推荐理由
- 生成首页兼排行榜结果

### 阶段 3：页面展示

- 首页兼排行榜页
- 偏好设置页
- 城市详情页
- 城市对比页
- 历史分析页

### 阶段 4：稳定性和答辩优化

- 异常处理
- 空数据兜底
- 数据更新时间展示
- README
- 报告材料

---

## 10. 遇到歧义时的处理规则

### 10.1 可以自行决策的情况

以下情况可以直接选最简单方案：

- 图表具体颜色和样式
- 某个页面是用柱状图还是折线图
- 某个函数拆分方式
- 首页兼排行榜页内部的布局细节

### 10.2 必须先停下来确认的情况

以下情况不要默认拍脑袋决定：

- 新增需求边界外功能
- 更换主要技术栈
- 修改城市名单
- 将 2 城市对比扩成多城市
- 将历史月度统计扩成全量日级体系
- 引入额外数据库或大数据平台
- 将 AQI 从增强项改成硬依赖，或反过来

---

## 11. 质量底线

以下标准属于最低质量线：

- 代码模块清晰，不能把所有逻辑堆在一个文件
- 爬虫、清洗、评分、展示逻辑必须分开
- 配置项集中管理，不能在多个地方重复写死
- 页面出现空数据时必须有提示
- 抓取失败时必须有错误信息
- 偏好改变后必须能看到结果变化
- AQI 不可用时必须明确显示当前为无 AQI 模式
- 页面打开时不依赖实时抓取才能显示结果
- 页面必须能显示最近更新时间

---

## 12. 什么算完成，什么不算完成

### 12.1 视为完成

- 有真实数据链路
- 有真实算法输出
- 有真实页面交互
- 有可说明的数据来源和字段说明
- 有明确的降级策略
- 工程目录结构符合已定义边界
- 页面默认读取本地持久化数据

### 12.2 不视为完成

- 只有静态页面，没有真实数据
- 只有天气展示，没有推荐逻辑
- 偏好页面只是摆设，不影响结果
- 历史分析只是写几段文字
- 页面能看，但老师无法解释数据从哪里来
- 用调试样例数据代替正式结果且未明确标识
- 每次访问页面都重新抓取外部网页

---

## 13. 后续协作要求

后续每次开始新阶段时，都应该先检查：

- 当前工作是否仍在 `requirement.md` 范围内
- 当前交付是否能在 `checklist.md` 中找到对应验收项
- 当前方案是否有更简单、更稳定的替代实现
- 当前文件是否放在正确的目录边界内

如果答案是否定的，先改方案，不要直接往下写。
